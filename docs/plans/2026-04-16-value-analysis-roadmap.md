# 价值分析功能路线图

## 背景

当前项目已经具备以下能力：

- 实时行情与舆情看板
- 深度分析页：历史分位、F-Score、估值结论层、多模型估值、Investment Thesis
- 历史回测页
- 财务溯源页：现金流质量、资本配置、经营稳定性、股东人数结构

这些能力已经能支持“单股基础研究”，但从完整的价值投资研究链条看，仍然缺少几个关键层：

- 行业与同行锚定
- 归一化盈利能力
- 资产负债表风险穿透
- 治理与管理层质量
- 业务理解与护城河来源
- 情景估值与下行测算

本路线图的目标，是把项目从“财务分析终端”推进到“价值研究终端”。

## 设计原则

- 先补研究链上的硬缺口，再补锦上添花
- 先做可落地、可验证的数据层，再做主观解释层
- 每个模块都要求：
  - 有独立后端接口或结构化字段
  - 有前端展示区块
  - 有基本测试
  - 有明确数据不足时的降级行为
- 每一轮优先做一个完整模块，不拆成半成品

## 开发顺序

推荐顺序：

1. 行业与同行对比
2. 归一化盈利
3. 资产负债表风险页
4. 治理与管理层质量
5. 业务理解层
6. 情景估值与下行测算

原因：

- 前 4 项对估值结论和仓位判断影响最大
- 前 3 项更偏结构化数据，适合先做
- 治理和业务理解更需要解释层，适合在数据骨架稳定后推进
- 情景估值应建立在前面几层都相对完整之后

## P0：核心缺口

### P0-1 行业与同行对比

目标：

- 给单股估值和财务判断一个横向锚点，避免“只和自己比”

建议能力：

- 每只股票支持维护同行池
- 展示同行估值横向对比：
  - PE
  - PB
  - EV/EBIT
  - EV/FCF
  - 股息率
  - ROE
  - ROIC 代理
- 输出行业中位数、龙头溢价、当前标的相对分位
- 支持在深度分析页增加“同行估值带”

后端需求：

- 增加股票到行业、同行池的映射
- 补充 EV/EBIT、EV/FCF 所需字段
- 提供同行汇总接口和缓存

前端落点：

- 深度分析页新增“行业与同行锚点”
- 对比分析页支持一键带入同行池

验收标准：

- 任一股票可看到至少 3 家同行对比
- 可比数据不足时，页面明确提示而不是报错
- 指标排序和中位值计算有测试覆盖

### P0-2 归一化盈利

目标：

- 解决周期高点利润、一次性损益、异常年份导致的估值失真

建议能力：

- normalized EPS
- normalized FCF
- 近 3 到 5 年中周期利润
- 一次性损益剔除提示
- “当前利润是否偏离中枢”的标签

后端需求：

- 对近几年利润、现金流、毛利率、费用率做平滑处理
- 增加归一化结果写入估值模型输入
- 允许多模型估值切换“报表利润 / 归一利润”

前端落点：

- 深度分析页估值结论层增加“归一化口径”
- 历史回测页可切换原始口径与归一化口径

验收标准：

- 多模型估值至少有一个模块可以切到归一化口径
- 对明显异常年份能给出解释标签
- 归一化逻辑有单元测试

### P0-3 资产负债表风险页

目标：

- 把价值陷阱识别从利润表延伸到资产负债表

建议能力：

- 短债覆盖率
- 现金 / 有息负债
- 利息保障倍数
- 应收、存货、预付异常增长
- 商誉与减值风险
- 资产负债表风险标签

后端需求：

- 补充资产负债表的关键字段抽取
- 输出风险因子和异常波动识别
- 形成独立 summary

前端落点：

- 财务溯源页增加“资产负债表风险”区块
- 深度分析页 Thesis 风险清单接入风险标签

验收标准：

- 能输出至少 5 个风险指标
- 异常值不会导致整页失败
- 服务层和接口层有测试

### P0-4 治理与管理层质量

目标：

- 让“值不值得长期持有”不只依赖财务数字

建议能力：

- 管理层持股与增减持
- 股权质押
- 分红、回购、再融资记录
- 审计意见
- 关联交易、诉讼与治理风险标签

后端需求：

- 接入治理相关数据源
- 建立事件分类和风险标签
- 形成治理画像 summary

前端落点：

- 深度分析页新增“治理与股东友好度”
- 财务溯源页信号面板可接入关键治理标签

验收标准：

- 能形成治理正向/中性/风险标签
- 风险事件有时间线
- 数据源不可用时能降级

## P1：研究深度增强

### P1-1 业务理解层

目标：

- 让系统从“看财务”走向“看生意”

建议能力：

- 分业务收入 / 利润结构
- 区域结构
- 产品结构
- 渠道结构
- 护城河来源标签：
  - 品牌
  - 成本
  - 渠道
  - 网络效应
  - 转换成本

前端落点：

- 深度分析页新增“业务结构与护城河”

### P1-2 情景估值与下行测算

目标：

- 把单点估值判断升级为分布式判断

建议能力：

- Bear / Base / Bull 三情景
- ROE、增长、分红、估值倍数敏感性矩阵
- 下行保护测算
- 预期收益区间而非单点

前端落点：

- 深度分析页新增“情景估值矩阵”

## P2：进阶能力

### P2-1 组合研究层

说明：

- 当前项目曾试过组合总览，但用户不需要首页组合面板
- 后续若重启组合层，不应该是“首页卡片”，而应该是独立研究模块

建议能力：

- 多标的研究池
- 候选池打分
- 行业暴露
- 组合层安全边际
- 组合情景回撤

### P2-2 研究工作流

建议能力：

- Thesis 版本管理
- 季报后自动复核
- 红旗事件提醒
- 自定义观察清单

## 模块依赖

- 行业与同行对比：基本独立，可优先开工
- 归一化盈利：依赖现有财务数据，可直接接在深度分析后
- 资产负债表风险页：依赖报表字段扩充
- 治理与管理层质量：依赖新增数据源，复杂度较高
- 业务理解层：依赖更丰富的结构化业务数据
- 情景估值：依赖归一化盈利和行业锚定，建议后做

## 建议的首个开发项

建议先做：`P0-1 行业与同行对比`

原因：

- 最能直接提升当前估值结论层的可信度
- 对用户理解“便宜还是贵”最直观
- 数据和 UI 都相对容易模块化
- 可以自然复用到深度分析页和对比分析页

## 开发节奏建议

- 每次只推进一个模块
- 每个模块按以下顺序交付：
  1. 后端数据结构
  2. 服务层与缓存
  3. 接口
  4. 前端展示
  5. 测试
  6. README 或路线图状态更新
